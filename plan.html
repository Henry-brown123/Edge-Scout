BETTING INTELLIGENCE PLATFORM — PROJECT PLAN
Core thesis
The betting market is most efficient in the most watched competitions. The Premier League gets priced by teams of quant analysts. Lower divisions, less-covered competitions, and non-outcome markets (goals, cards, corners) receive far less sophisticated pricing. A well-calibrated model with good historical data finds genuine edge more consistently in these markets than in Premier League match outcomes. The strategy is to start where the model is strongest — major leagues, match outcomes — and systematically expand into less efficient markets as data and confidence grows. Simultaneously, the bookmaker management module ensures the operation remains sustainable — spreading activity across 20+ operators and exchanges so no single restriction event materially impacts returns.
The ultimate goal is a self-improving, self-managing betting intelligence system that gets smarter with every match played and more operationally resilient with every bookmaker added.
Key principles
Model logic and strategy discussions happen in Claude chat. Implementation happens in Claude Code.
No real money until paper trading demonstrates genuine edge — positive ROI after 8 weeks AND calibration within ±5pp across all bands.
No model changes while backfills are running. One clear brief at a time to Claude Code, wait for completion before the next.
Every resolved fixture is training data — the model gets smarter continuously.
Buy data when it demonstrably improves the model's edge, not when the model is already profitable. Waiting until profitable to buy the tools that make you profitable is a false constraint.
Account management is as important as model quality — a profitable model with restricted accounts generates no returns.
Sustainable long-term operation beats maximum short-term extraction.
The model must be team-specific, not just league-specific. Team profiles override league averages as data accumulates.
Build properly, not reactively. Agree design in Claude chat first. Claude Code gets a complete specification, not a starting point for improvisation.
No patching. If a fix touches more than one file or core infrastructure, design discussion in Claude chat first.
Define done before starting. Every brief must specify what success looks like — tests, outputs, endpoints to verify.
Infrastructure is the foundation. Server, backfill, cron, and data integrity must be solid before adding features on top.
